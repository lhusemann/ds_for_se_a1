\section{Introduction}
% ==========================================
\textit{[Define motivation and objectives. Substantiate the problem context with established literature (e.g., \cite{Ref1}). Formulate the central hypothesis or research goal.]}
\begin{changedblock}
As software systems evolve, their architectural documentation frequently deviates from the actual implementation. To maintain system integrity and facilitate comprehension, automated architecture recovery techniques are essential. This study evaluates the efficacy of structural clustering algorithms and Large Language Models (LLMs) in extracting and describing the architectural components of an existing software system.

Specifically, this research focuses on the \textbf{Codecs} component of the \textbf{Apache Lucene} project. By utilizing the \textbf{ARCADE} tool suite, we extract system-wide dependencies and isolate component-relevant relationships to generate architectural clusters. The objective is to assess the performance of three established clustering algorithms---\textbf{WCA, Limbo, and ACDC}---measuring their structural accuracy through \textbf{a2a} (Architecture-to-Architecture) and \textbf{Cvg} (Cluster Coverage) metrics.

Furthermore, this study investigates the intersection of structural recovery and generative AI. We evaluate how different prompting techniques influence an LLM's ability to synthesize high-level architectural descriptions from source code. This evaluation utilizes a dual-tier approach: local prototyping with the lightweight \texttt{Mistral-7B-Instruct-v0.3} and production-scale execution with the heavyweight \texttt{Mistral-Large-3-675B-Instruct-2512}.

Ultimately, this work aims to address the variance in accuracy across different clustering algorithms and determine the impact of prompting strategies on the quality of AI-generated architectural summaries.
\end{changedblock}

\section{Study Design}
% ==========================================

\subsection{Research Questions}
\textit{[Motivation for the individual research questions. Why are these metrics/aspects relevant?]}
\begin{itemize}
    \item \textbf{RQ1:} How do different clustering algorithms vary in their ability to accurately determine the architectural components of a system?
    \item \textbf{RQ2:} How do different prompting techniques impact LLM's ability to accurately describe architectural components based on source code?
\end{itemize}

\subsection{Methodology}
\textit{[Architecture of the experimental setup. Specify the data collection process (e.g., scraping, APIs, existing corpora).]}

\begin{changedblock}
\noindent The experimental setup follows a phased workflow designed to recover the architecture of the Apache Lucene Codecs component through a combination of structural clustering and Large Language Model (LLM) analysis.

\subsubsection{Dependency Extraction and Preprocessing}
The initial phase involves compiling the Java source code to generate compiled *.jar files. System-wide dependencies are extracted using the ARCADE JavaParser tool, which produces a master .rsf file representing all project relationships. To isolate the scope to the \textit{Codecs} component, the dependencies are filtered such that only relationships where the source or target packages reside within the component’s namespace are retained.

\subsubsection{Architectural Clustering Algorithms}
To address RQ1, three distinct structural clustering algorithms are applied to the filtered dependency graph:
\begin{itemize}
    \item \textbf{Weighted Combined Algorithm (WCA):} A hierarchical clustering approach utilizing edge weights to determine cluster merges.
    \item \textbf{LIMBO:} An information-theoretic algorithm designed to minimize the loss of mutual information during clustering.
    \item \textbf{ACDC:} A heuristic-based algorithm that identifies architectural patterns based on structural connectivity.
\end{itemize}
Technical implementation details and CLI commands are provided in Appendix \ref{app:commands}.

\subsubsection{LLM-Based Architectural Recovery}
\noindent To address RQ2, the study implements a dual-stage pipeline designed to verify prompt stability before scaling to the full dataset.  \paragraph{Prototyping Tier: Lightweight Evaluation}
The initial prototyping phase is conducted using the assigned lightweight model, \texttt{Mistral-7B-Instruct-v0.3}. This model is an instruction-fine-tuned version of the Mistral-7B-v0.3 base model, optimized for conversational tasks and complex direction-following. Key technical features utilized during this phase include:
\begin{itemize}
\item \textbf{Environment:} Prototyping is executed within Google Colab to verify prompt logic stability on smaller samples before full deployment.
\item \textbf{Technical Capabilities:} The model's native support for function calling and the specialized Mistral tokenizer are leveraged to handle structural architectural data efficiently.
\end{itemize}  
\paragraph{Production Tier: Heavyweight Scaling}
Upon validation of the prompting logic, the recovery process is scaled to the university's High Performance Computing (HPC) cluster.
\begin{itemize}
\item \textbf{Model Scaling:} The study utilizes the heavyweight \texttt{Mistral-Large-3-675B-Instruct-2512} to process the complete Codecs component of the Lucene project.
\item \textbf{Execution Framework:} Scaling is managed via Slurm job scheduling.
\item \textbf{Output Generation:} The primary goal of this tier is to generate concise architectural titles and a high-level descriptive summary for each cluster identified in the structural recovery phase.
\end{itemize}  
\end{changedblock}

\subsection{Data Analysis Process}
\textit{[Detailed explanation of the analysis pipeline. Describe applied algorithms, statistical methods, heuristics, or machine learning models. Ensure reproducibility.]}\\

\begin{changedblock}
The data analysis pipeline is structured into three primary phases: structural normalization, quantitative statistical profiling, and comparative similarity evaluation. 
 
\subsubsection{Structural Normalization and Metadata Extraction}
Raw architectural outputs from the ARCADE toolset are generated in the \textit{.rsf} format, which represents dependencies line-by-line as \texttt{depends Source\_Package Target\_Package}. To facilitate objective comparison, a post-processing utility is utilized to transform these raw files into structured plain-text artifacts. This utility performs the following operations:
\begin{itemize}
\item \textbf{Sorting:} Clusters are ordered by cardinality (size), and internal elements are sorted alphabetically to ensure deterministic comparison.
\item \textbf{Metadata Generation:} For each algorithmic run, the utility extracts a statistical profile used to determine the optimality of the clustering parameters.
\end{itemize}  \subsubsection{Quantitative Profiling Metrics}Algorithm selection and parameter tuning are based on a set of distribution metrics that characterize the resulting architecture. These metrics include:\begin{itemize}\item \textbf{Cluster Cardinality:} Total number of identified clusters and the count of singletons (size = 1) versus multi-element clusters (size > 1).\item \textbf{Central Tendency:} Average size per cluster across the entire system and specifically for clusters with size $> 1$.\item \textbf{Distribution Variance:} Median size for multi-element clusters and the absolute size of the largest identified cluster.\end{itemize}\subsubsection{Comparative Similarity Evaluation (RQ1)}
To quantify the variance between clustering algorithms, the processed \textit{.rsf} files are analyzed using the \textbf{Architecture-to-Architecture (a2a)} and \textbf{Cluster Coverage (cvg)} metrics.
\begin{itemize}
\item \textbf{a2a Similarity:} Measures the global structural distance between two architectures, providing a percentage-based similarity score.
\item \textbf{Cluster Coverage (cvg):} An asymmetric metric measuring the extent to which clusters in one architecture overlap with clusters in another.
\end{itemize}
A high percentage in these metrics indicates a high degree of structural convergence between different algorithmic approaches.  \subsubsection{Qualitative LLM Assessment (RQ2)}
For the evaluation of LLM-generated architectural descriptions, a qualitative assessment is conducted. This involves mapping the generated titles and high-level summaries back to the source code elements within each cluster to evaluate semantic accuracy and the impact of different prompting techniques. \\

\section{Discussion}
% ==========================================
\textit{[Interpretation of the results from Section 3. Comparison with the initial hypothesis.]}

\begin{changedblock}
This section interprets the quantitative findings from the structural recovery phase and evaluates the efficacy of the applied clustering algorithms in accurately determining the architectural components of the Apache Lucene Codecs module.

\subsection{Interpretation of Clustering Profiles (RQ1)}
An analysis of the clustering profiles indicates that the WCA algorithm, particularly when utilizing the UEMNM measure, yields a more balanced and structurally cohesive architecture for the given target of 70 clusters. WCA (UEMNM) produces a higher median cluster size (6) and a reasonable maximum cluster size (11), suggesting it successfully identifies moderately sized, functional subsystems within the code.

Conversely, as evidenced by the distribution metrics in Table \ref{tab:cluster_stats}, LIMBO demonstrates an aggressive 'bottom-up' tendency, resulting in a significantly skewed architecture. While targeting the same total cluster count, LIMBO aggregates a massive number of elements into a single monolithic cluster (size 41) while simultaneously generating a high volume of extremely small clusters (median size 2). Qualitative inspection of the elements within this monolith reveals that LIMBO frequently groups structurally unrelated components into this single large cluster, rendering its output highly suboptimal for practical architecture recovery in this specific context.

\subsection{Analysis of Structural Similarity and Overlap}
As detailed in Table \ref{tab:similarity_metrics}, the Architecture-to-Architecture evaluation reveals the highest structural similarity (83.30\%) between LIMBO and WCA. In contrast, ACDC has a lower overall similarity score.

However, the global similarity masks severe local discrepancies, which are exposed by the Cluster Coverage metric. The inherent asymmetry of cvg is evident in the comparison between WCA and ACDC (20.0\% vs. 2.0\% in Table \ref{tab:similarity_metrics}), a variance heavily influenced by the differing total cluster counts. Most notably, the coverage between LIMBO and ACDC registers at exactly 0.0\%. This complete lack of structural overlap is a direct byproduct of LIMBO’s highly skewed distribution; the 41-element monolith (refer back to Table \ref{tab:cluster_stats}) fundamentally misaligns with ACDC's heuristic patterns, failing to meet the minimum percentage thresholds required by the ARCADE cvg tool to register a match.
\end{changedblock}
